Section 01

Crypto? Like dogecoin?

The Open-Channel Problem

On the internet, it is very hard to ensure that information traveling between two individuals can be read by them only.

Sniffing, Recording, Man in The Middle... So many attack vectors exist to collect information about conversations between two individuals. Cryptography's role is to create protocols and algorithms to ensure the safety of a conversation between our subjects: Alice & Bob.

While Eve eavesdrops, Alice and Bob want to communicate privately
[!] Key Idea
Cryptography: the science of transforming messages so only authorized parties can understand them.
[*] Crypto is "Open Source"
Unlike most CTF categories (which are glorified guesswork), crypto is built on Kerckhoffs's principle: the attacker knows everything about the system, except the key. Closed source cryptography challenges are called rev, and you should flee from them like the plague! Your goal in a crypto CTF is to find what went wrong in a known system: a weak key, a reused nonce, a broken mode... You always know how it works; you just need to figure out what's broken.

Section 02

Caesar Cipher and Frequency Analysis

The Caesar Cipher is probably one of the oldest, most intuitive, and worst cipher that exists. Each letter is shifted by a fixed number $n$. Julius Caesar used a shift of 3:
A -> D,
B -> E,
C -> F...

Caesar Encryption
$$E(x) = (x + k) \mod 26$$ $$D(x) = (x - k) \mod 26$$

The Caesar Cipher, due to its cyclic nature, can be represented as a wheel (below). In modern days, the Cipher is more known as ROT-n, with n being the shift value.

[*] Fun fact
Old forums had a habit of hiding "spoiler" messages using ROT-13. References to this habit can still be found in modern media.

Exploit Vector: Frequency Analysis

The Caesar cipher is known as a mono-alphabetic cipher: each letter always maps to the same ciphertext letter, and as such, the frequency distribution is preserved. In English, the letter "E" appears ~12.7% of the time. If we see a letter appearing ~12.7% in the ciphertext, we can deduce it is likely E!

[!] Weakness
Any mono-alphabetic substitution cipher (not just Caesar) is vulnerable to frequency analysis. The Caesar Cipher however is especially weak: the key space (number of different values the secret key can be) is $\leq 26$, trivially brute-forceable!

Section 03

Vigenere Cipher and Kasiski Test

The Vigenere cipher improves on Caesar by using a keyword instead of a single shift. Each letter of the plaintext is shifted by the corresponding letter of the key (repeated as many times as needed).

Vigenere Encryption
$$C_i = (P_i + K_{i \bmod |K|}) \mod 26$$

Step-by-Step Animation

Position01234567891011
PlaintextATTACKATDAWN
Key LEMON LEMON LE
Ciphertext
Key "LEMON" repeats over the plaintext, each letter gets a different shift

Any letter of the ciphertext is the result of both the length of the key and letter of the original message. As such, two same letters of the plaintext do not necessarily produce the same output:simple frequency analysis fails

Attack Vector: Kasiski Test

But Vigenere isn't unbreakable (at least not since the 1850s). If the same plaintext sequence aligns with the same key position, it produces identical ciphertext: the function is isomorphic. The Kasiski test finds these repeats:

[*] Attack Strategy
  1. Find repeated sequences in the ciphertext
  2. Compute GCD $n$ of all distances -> probable key length
  3. Split ciphertext into groups (one per key position)
  4. Apply frequency analysis on each group independently / Bruteforce keys is $n$ is small enough

Section 04

One-Time Pad (OTP)

The weakness of Vigenere was the short, repeating key. What if we make the key as long as the message itself, thus never re-using it?

Well, you get the OTP Cipher, which now resists frequency analysis and key patterns. The message is now as secure as the key is.

Vigenere (short key)

Plaintext: ATTACKATDAWN
Key:       LEMONLEMONLE
^ key repeats -> pattern leaks

OTP (full-length key)

Plaintext: ATTACKATDAWN
Key:       XMCKPQSUATBG
^ truly random, same length -> no attack vectors
One-Time Pad
$$C_i = P_i \oplus K_i \quad\text{where}\quad |K| = |P|$$
Key K is truly random and used exactly once
[!] Information-Theoretic Security
The OTP has been mathematically proven to be unbreakable (Shannon, 1949). Given any ciphertext, every possible plaintext is equally likely without the key. It's the gold standard of encryption.
[!] The Catch
  • Key must be truly random (not pseudorandom)
  • Key must be at least as long as the message
  • Key must never be reused
  • Key must be generated over a charset superset of the key's
  • Key must be shared securely
... wait a second. For the procedure to work, the key must be shared securely. But if the key can be shared securely, why not share the message directly?

Despite the theoretical security of the OTP, it remains very hard to implement on its own due to the assumption of mutual key knowledge between pairs.

Section 05

AES - Symmetric Encryption

OTP is impractical, but brings a good theoretical framework to introduce symmetric-key ciphers, which expand a short key into secure encryption. AES (Advanced Encryption Standard) is the modern standard, used pretty much everywhere from HTTPS to disk encryption.

AES is said to be symmetric: the same key encrypts and decrypts. Think of it like a "lock" being opened and closed.
[*] Shannon's Principles
To prevent statistical analysis of symmetrical ciphers, two principles are applied:

Confusion: each part of ciphertext should depend on several parts of the key, making the relationship between key and ciphertext complex. Any small change in the key results in considerable changes of the ciphertext.

Diffusion: changing a single plaintext bit should affect many ciphertext bits, spreading the input's statistical structure across the output. Any small change in the plaintext results in considerable changes of the ciphertext.

AES Round - 4 Steps

AES processes data in blocks of fixed bit size. Each round applies 4 transformations. AES-128 uses 10 rounds over 128-bit blocks.

AES Round
1. SubBytes (confusion)
Each byte replaced via a lookup table (S-Box) - adds non-linearity
byte 0x53 -> S-Box[5][3] = 0xED
|
2. ShiftRows (diffusion)
Rows of the state matrix are cyclically shifted left
Row 0: no shift | Row 1: shift 1 | Row 2: shift 2 | Row 3: shift 3
|
3. MixColumns (diffusion)
Columns mixed via matrix multiplication in GF(2^8) - each output byte depends on all 4 input bytes
|
4. AddRoundKey (key mixing)
XOR state with the round key derived from the main key
state[i] = state[i] XOR roundKey[i] for each byte

Section 06

AES CTF Pitfalls

In CTF challenges, AES implementations often have deliberate weaknesses. Below is a list of the most common ones to look for. Since this is an introduction, we will not go over each of these, but you are greatly encouraged to do so during your own time :p

1.
ECB Mode
Identical plaintext blocks -> identical ciphertext blocks. Swap, duplicate, or analyze blocks to break the scheme.
2.
IV Reuse (CBC)
Reusing the initialization vector with the same key leaks XOR of two plaintexts: $C_1 \oplus C_2 = P_1 \oplus P_2$
3.
Padding Oracle
Server reveals whether padding is valid/invalid. By manipulating ciphertext bytes, decrypt block by block without the key.
4.
Bit Flipping (CBC/CTR)
Flipping bit $i$ in ciphertext block $n$ flips the same bit in plaintext block $n+1$. This allows to modify data without knowing the key and produce valid ciphers without knowledge of the key.
5.
Weak Key Derivation
Key derived from short password, timestamp, or predictable seed. Brute-force the key space instead of attacking AES itself.
6.
Known Plaintext
If you know part of the plaintext (usually headers or file magic bytes), use it to recover key material or attack the mode.
[!] CTF Tip
When you see AES in a CTF, first identify the mode (ECB/CBC/CTR/GCM...) and look for implementation mistakes. If the implementation and/or mode are custom, think about what has changed compared to a known mode, and what this could imply!
[*] Fun fact
The attack vector 6. is close to what was used by Alan Turing and his team during WW2 to break the enigma machine! In this context, small knowledge of the plaintext drastically reduces the entropy of the cipher.
Section 07

The Key Sharing Problem... again

The Problem

Symmetric encryption, much like OTP, works great! Efficient and secure designs which work over keys of regular size, but only if both parties already share a key. But how does Alice send Bob the key in the first place?

Sending a key over an open channel defeats the purpose, since it becomes public knowledge instantly. Encrypting the key using another call to a symmetric cipher results in the same problem, but with a different key now.

The Lockbox Analogy

Let's go back to the previous lockbox analogy. Except now, instead of using the same padlock, we follow a different protocol:

  • Bob generates both his own personal private key and public padlock
  • Bob can safely send the padlock over an open channel to Alice: the padlock can be locked, but without the key it can't be unlocked.
  • Alice uses Bob's padlock to secure the box in which she hid a message
  • The box with the message being secured, it can be safely shared over an open channel
  • Only Bob has access to the key, so only he can open the locked box
[!] Key Insight
Asymmetrical Ciphers use two keys: a public key (which can be shared freely) and a private key (which is kept secret). The public key is only used to encrypt data, knowing it doesn't reveal the private key.
Section 08

RSA

RSA (Rivest-Shamir-Adleman) is one of the most famous asymmetric algorithms. Its security relies on the difficulty of factoring large numbers.

Key Generation

1 / 5
1. Choose two large primes
$$p = 61, \quad q = 53$$
In practice, p and q are hundreds of digits long
2. Compute n and phi(n)
$$n = p \times q = 61 \times 53 = 3233$$ $$\phi(n) = (p-1)(q-1) = 60 \times 52 = 3120$$
n is the modulus (public). phi(n) is Euler's totient (secret).
3. Choose public exponent e
$$e = 17 \quad \text{where} \quad \gcd(e, \phi(n)) = 1$$
e must be coprime with phi(n). Common choice: 65537
4. Compute private exponent d
$$d = e^{-1} \mod \phi(n) = 17^{-1} \mod 3120 = 2753$$ $$e \times d \equiv 1 \pmod{\phi(n)}$$
d is the modular inverse of e (extended Euclidean algorithm)
Result - Key Pair
PUBLIC KEY
$(e, n) = (17, 3233)$
PRIVATE KEY
$(d, n) = (2753, 3233)$

Encrypt and Decrypt

$$c = m^e \mod n$$
Encryption
$$m = c^d \mod n$$
Decryption

Why Is RSA Secure?

Given two primes $p$ and $q$, it is trivially easy to compute their product $n=p \times q$.

However, the inverse operation is very hard! Given only $n$, we do not know any efficient method to find the numbers $p$ and $q$ such that their product is equal to $n$.

[!] Prime trapdoor Function
By ensuring that $p$ and $q$ are prime, we can assert that no other integer pair can have their product equal to $n$. With very large values that have no apparent relation, it becomes cryptographically secure.
Section 09

RSA CTF Pitfalls

Like AES, RSA CTF exploit implementation mistakes, they don't break the algorithm itself.

1.
Small Primes
If $p$ or $q$ are too small, $n$ can be factored easily. Tools like factordb or yafu could break it in seconds.
2.
Small exponent $e$
If $m^e < n$, then $c = m^e$ and you can just take the $e$-th root! This also enables Coppersmith's attack with partial knowledge.
3.
Common Modulus
If a same message is encrypted with the same $n$ but different exponents $e$, we may use the extended GCD algorithm to recover plaintext without factoring.
4.
Wiener's Attack
When $d$ is small ($d < n^{0.25}/3$), continued fractions on $e/n$ could reveal the private key $d$ directly.
5.
Shared Prime ($p$)
If two RSA keys share a prime factor, we may compute $\text{GCD}(n_1, n_2)$ to reveal the shared prime $p$, thus allowing to deduce $q=n/p$
6.
Hastad's Broadcast
If the same message is sent to $\geq e$ recipients (especially with small values of $e$), the Chinese Remainder Theorem + $e$-th root method could recover $m$.
[!] CTF Tip
The first reflexes when encountering an RSA implementation is to check those common details! Is $n$ factorable? Is $e$ suspiciously small/large? Are there multiple ciphertexts with the same $n$? Does the implementation give information about the padding correctness? Start with factordb.com and RsaCtfTool.

Section 11

PRNG vs CSPRNG

Many crypto systems need random numbers. To generate the RSA-primes, unique OTP keys, high entropy passwords... But not all "random" is created equal.

PRNG

Pseudo-Random Number Generator.
A set of deterministic algorithms that looks random, but always produces the same values in the same order over an initial state seed and a relation $f$ of set parameters.

Examples: LCG, Mersenne Twister, LFSR, Middle Square

CSPRNG

Cryptographically Secure PRNG. A set of methods with an output that is computationally indistinguishable from true randomness. Such algorithms are necessary for cryptographic assets such as keys, IVs, and nonces.

/dev/urandom uses a pool of data from disk usage, cpu activity, user inputs and other devices of high entropy to generate CSPRNG values

Linear Congruential Generator (LCG)

One of the simplest PRNGs is the LCG. Given the parameters $a, c, m$ and one value $X_k$, you can predict every value $X_{\geq k}$:

LCG Formula
$$X_{n+1} = (a \cdot X_n + c) \mod m$$
The first value of the cycle $X_0$ is called the seed

LCG Scatter and Patterns

Due to the linear relationship between consecutive LCG pairs $(X_n, X_{n+1})$, we can observe visible patterns when plotted (by Marsaglia's theorem)

[!] CTF Takeaway
If a CTF challenge uses a PRNG (especially LCG or Mersenne Twister) for key generation or encryption, the randomness is breakable. Recover the state -> predict future outputs -> break the crypto.
